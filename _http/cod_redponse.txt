
Інформаційні		
100	
	Continue (Продовжуй) Сервер задоволений початковими відомостями про запит, клієнт може продовжувати пересилати заголовки. З'явився в версії HTTP / 1.1.
101		
	Switching Protocol (Перемикання протоколів)	Сервер пропонує перейти на більш відповідний для зазначеного ресурсу протокол; список пропонованих протоколів сервер обов'язково вказує в поле заголовка Upgrade. Якщо клієнта це зацікавить, то він посилає новий запит із зазначенням іншого протоколу.
102		
	Processing (Йде обробка)	Запит прийнятий сервером, але на його обробку знадобиться тривалий час. Ця відповідь використовується для того, щоб клієнт не розірвав з'єднання через перевищення часу очікування. Клієнт при отриманні такої відповіді повинен скинути таймер і чекати наступної команди в звичайному режимі. З'явився в WebDAV.
Успішні		
200	ОК Успішно оброблений запит. «Успішність» залежить від методу HTTP, який був запитаний:
		GET: Запитаний ресурс був знайдений і переданий в тілі відповіді.
		HEAD: Заголовок переданий у відповіді.
		POST: Ресурс, що описує результат дії сервера на запит, переданий в тілі відповіді.
		TRACE: Тіло відповіді містить тіло запиту отриманого сервером.
201	Created (Створено) Запит успішно виконаний і в результаті був створений новий ресурс. Цей код зазвичай надсилається у відповідь на запит PUT «помістити».
202	Accepted (Прийнято) Запит прийнятий, але ще не оброблений. Клієнту не обов'язково чекати остаточної передачі повідомлення, так як може бути розпочато дуже довгий процес.
203	Non-Authoritative Information (Інформація не авторитетна) Цей код відповіді означає, що інформація, яка була повернута, була надана не сервером, а іншим джерелом. Аналогічно відповіді 200, але в цьому випадку передана інформація була взята не зі первинного джерела і тому може бути неактуальною.
204	No Content (Немає вмісту) Сервер успішно обробив запит, але у відповіді були передані тільки заголовки без тексту повідомлення. Клієнт не повинен оновлювати вміст документа, але може застосувати до нього отримані метадані. Клієнт може використовувати їх для поновлення кешованих заголовків для попередніх ресурсів.
205	Reset Content (Скинути вміст)	Цим кодом сервер зобов'язує клієнта скинути введені користувачем дані. Тіла повідомлення сервер при цьому не передає і документ оновлювати не обов'язково.
206	Partial Content (Частковий вміст) Сервер вдало виконав частковий GET-запит, повернувши тільки частина контенту. У заголовку Content-Range сервер вказує байтові діапазони вмісту. Особливу увагу при роботі з подібними відповідями слід приділити кешуванню.
Перенаправлення
300	Multiple Choice (Множинний вибір) Цей код відповіді надсилається, коли запит має більш ніж один з можливих відповідей (по типу MIME, за мовою або за іншими характеристиками). Сервер передає з повідомленням список альтернатив, даючи можливість зробити вибір клієнта автоматично або користувачеві.
301	Moved Permanently (Переміщений на постійній основі)	Цей код відповіді означає, що URI запитуваного ресурсу був змінений. Новий URI вказується в полі Location заголовка.
302	Found (Знайдено); Moved Temporarily (Переміщений тимчасово) Цей код відповіді означає, що запитаний ресурс тимчасово доступний по іншому URI, вказаною в поле Location заголовка.
303	See Other (Дивитися інше) Сервер відправив цей відповідь, щоб направити клієнта для отримання запитуваного ресурсу по іншому URI з використанням методу GET. Інший URI вказується в полі Location заголовка.
304	Not Modified (Не змінювалося)	Ця відповідь використовується для кешування. Він повідомляє клієнту, що відповідь не був змінений. Таким чином, клієнт може продовжувати використовувати ту ж кешовану версію відповіді. При цьому повідомлення сервера не повинно містити тіла.
305	Use Proxy (Використовуй проксі)	Запит до запитуваного ресурсу повинен здійснюватися через проксі-сервер, URI якого вказано в полі Location заголовка. Цей код відповіді не підтримується в основному з міркувань безпеки.
306	невикористаний	Цей код відповіді більше не використовується, в даний час зарезервовано.
307	Temporary Redirect (Тимчасова переадресація) Запитуваний ресурс на короткий час доступний по іншому URI, вказаною в поле Location заголовка. Має ту ж семантику, що і 302 Found код відповіді HTTP, за винятком того, що клієнт не повинен змінювати використовуваний HTTP-метод: якщо використовувався POST в першому запиті, він повинен використовувати POST і в другому запиті.
308	Permanent Redirect (Постійна переадресація) Це означає, що ресурс тепер постійно знаходиться за іншою URI, вказаною в поле Location заголовка. Має ту ж семантику, що і код відповіді HTTP 301 Moved Permanently, за винятком того, що клієнт не повинен змінювати використовуваний HTTP-метод: якщо використовувався POST в першому запиті, він повинен використовувати POST і в другому запиті.
помилки клієнта
400	Bad Request (Поганий запит) Ця відповідь означає, що сервер не зміг зрозуміти клієнтський запит через синтаксичну помилку.
401	Unauthorized (Не авторизований)	Для отримання запитаного відповіді потрібно аутентифікація. Це схоже на 403 відповідь, але в цьому випадку можлива аутентифікація. У заголовку відповідь повинен містити поле WWW-Authenticate з переліком умов аутентифікації.
402	Payment Required (Потрібно оплата) Цей код відповіді зарезервований для майбутнього використання. Початкова мета створення цього коду полягала у використанні його для цифрових платіжних систем, однак в даний час він не використовується.
403	Forbidden (Заборонено)	Сервер зрозумів запит, але він відмовляється його виконувати через обмеження в доступі для клієнта до зазначеного ресурсу. Клієнт не має прав доступу до контенту, тому сервер відмовляється дати правильну відповідь. Найбільш ймовірними причинами обмеження може служити спроба доступу до системних ресурсів веб-сервера (наприклад, файлів .htaccess або .htpasswd) або до файлів, доступ до яких був закритий за допомогою конфігураційних файлів.
404	Not Found (Не знайдено)	Сервер не може знайти запитуваний ресурс. Основна причина - помилка в написанні адреси Web-сторінки. Цей код відповіді, ймовірно, є найвідомішим з-за його поширеності в Інтернеті.
405	Method Not Allowed (Метод не підтримується)	Метод запиту відомий серверу, але був відключений і не може бути використаний. У відповіді сервер повинен вказати доступні методи в заголовку Allow, розділивши їх комою. Два обов'язкових методу: GET і HEAD ніколи не повинні бути відключені і не повинні повертати цей код помилки.
406	Not Acceptable (Неприпустимо)	Запитаний URI не може задовольнити переданим в заголовку характеристикам. Якщо метод не була HEAD, то сервер повинен повернути список допустимих характеристик для даного ресурсу.
407	Proxy Authentication Required (Потрібно аутентифікація проксі)	Ця відповідь схожий на 401, але тут перевірка справжності (аутентифікація) проводиться для проксі-сервера.
408	Request Timeout (зв 'язку)	Час очікування сервером передачі від клієнта минув. Це означає, що сервер хотів би відключити це неиспользуемое з'єднання. Зверніть увагу, що деякі сервери просто закривають з'єднання без відправки цього повідомлення.
409	Conflict (Конфлікт)	Ця відповідь буде відправлений, коли запит буде конфліктувати з поточним станом сервера. Таке можливо, наприклад, коли два клієнта намагаються змінити ресурс за допомогою методу PUT.
410	Gone (Вилучений)	Ця відповідь буде відправлений, коли запитуваний контент за вказаною URL був видалений з сервера. Сервера в цьому випадку невідомо і місце розташування альтернативного документа (наприклад, копії).
411	Length Required (Потрібно довжина)	Сервер відхилив запит, тому що поле заголовка Content-Length не визначене, а сервер вимагає його. Така відповідь природний для запитів типу POST і PUT. Наприклад, якщо за вказаною URI проводиться завантаження файлів, а на сервері стоїть обмеження на їх обсяг.
412	Precondition Failed (Умова помилково)	Клієнт вказав умовні поля в заголовках запиту (наприклад, If-Match і ін.), Які сервер не виконує.
413	Payload Too Large. Раніше - Request Entity Too Large (Запитаний занадто великий об'єкт)	Об'єкт запиту більше обмежень, які визначені на сервері. Сервер може закрити з'єднання, щоб припинити подальшу передачу запиту або повернути поле заголовка Retry-After із зазначенням часу, після закінчення якого можна повторити аналогічний запит.
414	URI Too Long. Раніше - Request-URI Too Long (Запитаний занадто великий URI)	Запитаний клієнтом URI занадто довгий для обробки сервером. Таку помилку можна спровокувати, наприклад, коли клієнт намагається передати довгі параметри через метод GET, а не POST.
415	Unsupported Media Type (Непідтримуваний тип даних)	Формат запитуваних типів даних не підтримується сервером, тому сервер відхиляє запит. З якихось причин сервер відмовляється працювати з вказаним типом даних при цьому методі.
416	Range Not Satisfiable. Раніше - Requested Range Not Satisfiable (Діапазон недосяжний)	Діапазон, вказаний полем заголовка Range в запиті, не може бути здійснений; можливо, що діапазон знаходиться поза розміру даних цільового URI і відсутній поле If-Range.
417	Expectation Failed (Помилка очікування)	Цей код відповіді означає, що очікування, вказане полем заголовка Expect запиту, не може бути виконано сервером.
421	Misdirected Request (Невірно адресований запит)	Запит був перенаправлений на сервер, не здатний дати відповідь.
451	Unavailable For Legal Reasons (Недоступно з юридичних причин)	Доступ до ресурсу закритий з юридичних причин
помилки сервера
500	Internal Server Error (Внутрішня помилка сервера)	Сервер зіткнувся з ситуацією, коли він не знає, як йому вчинити. Будь-яка внутрішня помилка сервера, яка не входить в рамки інших помилок класу.
501	Not Implemented (Не реалізовано)	Метод запиту не підтримується сервером і не може бути опрацьований. Типова відповідь для випадків, коли сервер не розуміє вказаний в запиті метод. Єдиними методами, які потрібні сервером для підтримки (і, отже, не повинні повертати цей код), є GET і HEAD.
502	Bad Gateway (Поганий шлюз)	Ця відповідь про помилку означає, що сервер, працюючи в якості шлюзу для отримання відповіді, необхідного для обробки запиту, отримав невірну відповідь. Сервер, виступаючи в ролі шлюзу або проксі-сервера, отримав недійсне відповідь було надіслане від висхідного сервера.
503	Service Unavailable (Сервіс недоступний)	Сервер не готовий обробляти запит. Сервер тимчасово не має можливості обробляти запити з технічних причин (обслуговування, перевантаження та інше). Зверніть увагу, що разом з цією відповіддю слід відправити зручну для користувача сторінку, яка пояснює проблему. Ці відповіді повинні використовуватися для тимчасових умов, і HTTP-заголовок Retry-After повинен, по можливості, містити розрахунковий час до відновлення обслуговування. Веб-майстер також повинен стежити за заголовками, пов'язаними з кешуванням, які відправляються разом з цією відповіддю, так як ці тимчасові відповіді зазвичай не кешуються.
504	Gateway Timeout (Шлюз не відповідає)	Ця відповідь про помилку надається, коли сервер діє як шлюз і не може отримати відповідь вчасно. Сервер в ролі шлюзу або проксі-сервера не дочекався відповіді від висхідного сервера для завершення поточного запиту.
505	HTTP Version Not Supported (Версія HTTP не підтримується)	Версія протоколу HTTP, яка використовується в запиті, що не підтримується сервером (або сервер відмовляється підтримувати зазначену версію).
509	Bandwidth Limit Exceeded (вичерпав пропускна ширина каналу)	Цей код використовується при перевищенні веб-майданчиком відведеного їй обмеження на споживання трафіка. В даному випадку власнику майданчика слід звернутися до свого хостинг-провайдеру. На даний момент цей код не описаний у жодному RFC і використовується тільки модулем «bw / limited», що входить в панель управління хостингом cPanel, де і був введений.
